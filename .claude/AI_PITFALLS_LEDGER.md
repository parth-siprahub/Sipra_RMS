SipraHub RMS: AI Pitfalls & Refactor Ledger
Mission: Document the transition from AI-generated "slop" to manually-refined, production-grade architecture.

[Entry 002] | Date: 2026-04-27
Pitfall: Identity Field Cross-Contamination. The ingestion logic utilized a "lazy fallback" where jira_username was mapped into the aws_email column when the primary email was missing from the source.

Refactor: (IN PROGRESS) Implementation of strict Pydantic validation on ingestion; enforcing aws_email as a mandatory, unique field and failing the record instead of defaulting to secondary identifiers.

The "Why": Business logic requires aws_email for DCLI billing reconciliation. Using JIRA names in an email field corrupts the primary key used for cross-referencing timesheets.

Learning: "Failing loudly" is better than "succeeding with bad data." AI-generated code tends to prioritize "making the script run to completion" over data schema integrity.

## [Entry 003] | Date: 2026-04-27
- **Pitfall:** **Direct-to-DB Bypass of Pydantic Validation.** In `employees/router.py` and `candidates/router.py`, the AI bypassed the `EmailStr` and slug validation by performing raw dictionary inserts into Supabase. 
- **Refactor:** Unified the onboarding logic into a single `employee_service.py` helper that enforces Pydantic validation and slug-format checks before any DB write.
- **The "Why":** Manual operators were entering Display Names ("Prakash KJ") into JIRA ID fields. Without a slug-format check (`no spaces`), the system corrupted the identity mapping used for billing.
- **Learning:** Never perform raw `supabase.table().insert()` inside a router. Always go through a service layer that validates the schema first.

## [Entry 004] | Date: 2026-04-27
- **Pitfall:** **Billing History Filtered by Transient Status Field.** Reports endpoints (`get_timesheet_comparison`, `get_compliance_report`, `calculate_billing`) filtered employees by `WHERE status = 'ACTIVE'`. This excluded all exited employees (Bashir, Kasir, Bindushree, Chandra) from April 2026 billing output, rendering the monthly report incomplete and the CSV export to Jaicind inaccurate.
- **Root Cause:** Status is a live, mutable field reflecting *current* state, not billing history. Using it as a filter in a *historical* query (April billing) conflates "who is active today" with "who was active during April." This is a category error in semantics.
- **Refactor:** Replace all three endpoints with a date-range filter: `(employees.start_date <= month_end) AND (employees.exit_date IS NULL OR employees.exit_date >= month_start)`. This includes any employee who worked any portion of the month, regardless of current status.
- **The "Why":** Billing history must be immutable and reference-accurate. An exited employee who worked April 1–22 must appear in April billing even if they're marked `EXITED` on April 27. Status changes must not retroactively erase history.
- **Learning:** Never use live operational state (status, is_active, is_deleted) to filter historical reports. Always use timestamps (start_date, exit_date, created_at) as the source of truth for "was this entity in scope for this period?"

## [Entry 005] | Date: 2026-04-27
- **Pitfall:** **"Human Middleware" Phase — Manual Email Field Corrections.** The ingestion pipeline had no intake validation for email field mapping. Corrupted records (siprahub.com emails in aws_email column) were force-pushed to production. Parth manually corrected them weekly, creating a hidden operational tax: unaudited, error-prone human edits to the billing identity anchor.
- **Root Cause:** No dead-letter queue (DLQ) or staging table. Failed validation errors were either dropped silently or force-pushed with bad data. No operator visibility into what failed and why.
- **Refactor:** Create `import_staging` table (DDL in Migration 014). Redirect all validation failures (422 errors, unmatched records, missing required fields) into the staging table with `error_reason` context. Provide an admin review UI to accept or reject each record. Disable force-push logic entirely.
- **The "Why":** Data integrity is not an iterative problem. Bad data in the billing identity anchor (emails) cascades into billing export, vendor reconciliation, and audit trails. Humans should never manually patch the database; the ingestion pipeline must validate and surface failures **before** write.
- **Learning:** Every data source ingestion must have three gates: (1) schema validation (Pydantic), (2) business logic validation (is this email actually valid for billing?), (3) a staging table (DLQ) for records that fail gates 1 or 2. Operator review, not silent deletion or force-push.

## [Entry 007] | Date: 2026-04-28
- **Pitfall:** **Single-Pass Matcher Gave Duplicate-Name Collisions.** In `EmployeeMatcher._match_employee_columns()`, the original loop checked `jira_username`, then `aws_email`, then `rms_name` in a single pass over employees. When two employees share the same `rms_name` (e.g., duplicate "Rajesh Sai Krishna V" rows — one EXITED with no jira_username, one ACTIVE with `jira_username='Rajesh Sai Krishna V'`), the EXITED employee at lower `id` was returned first via the `rms_name` fallback, before the ACTIVE employee could match on its explicit `jira_username`. Result: 70 April Jira rows were tagged to the wrong (EXITED) employee, yielding `jira_hours=0` for the ACTIVE employee and a false `NON_COMPLIANT` flag.
- **Root Cause:** Iteration order + a single loop meant the `rms_name` fallback could fire before the `jira_username` exact match on a later employee in the list. The correct semantics require all explicit-field matches across every employee to be exhausted *before* falling back to name matching.
- **Refactor:** Split `_match_employee_columns` into two passes: Pass 1 iterates all employees checking `jira_username`/`aws_email` (whichever is relevant); Pass 2 (only reached if Pass 1 found nothing) iterates all employees checking `rms_name`. This ensures an explicit identifier always beats a name coincidence, regardless of `id` ordering.
- **The "Why":** Identity resolution for billing must be deterministic and prefer authoritative fields. `jira_username` is explicit, operator-set, and unique-per-person. `rms_name` is a display string that can collide across duplicate/re-onboarded employees. Explicit field > display field is a non-negotiable precedence rule.
- **Learning:** Any multi-field lookup function that has a "fallback" field must exhaust the primary field across ALL candidates before applying the fallback. A single loop that checks primary-then-fallback per record inherits the iteration order as an invisible precedence rule — a hidden bug waiting for the first duplicate.

## [Entry 006] | Date: 2026-04-28
- **Pitfall:** **Hardcoded 176h Billing Target Applied to All Employees.** `calculate_billing()` applied the full-month `billing_config.billable_hours` (176h) uniformly to all employees, including those who started or exited mid-month. Employees like Pathak (exit 4/24) and Bindushree (start 4/2, exit 4/27) were evaluated against a target they could never legitimately hit, producing false "Non-compliant" flags and incorrect billing exports.
- **Root Cause:** The billing target was read once from `billing_config` and reused for every employee. No per-employee window calculation. The `176` (or config value) was an ambient constant, not a function of each employee's tenure within the billing period.
- **Refactor:** Created `backend/app/utils/date_utils.py` with `get_working_days(start, end)` (India 2026 holidays included) and `prorated_target_hours(month_start, month_end, emp_start, emp_exit)`. `calculate_billing()` now computes `billable_hours` per employee: `working_days(max(month_start, emp_start), min(month_end, emp_exit or today)) × 8`. Full-month employees are unaffected; mid-month exits receive accurate prorated targets.
- **The "Why":** A billing system that penalises an employee for hours they were contractually not present is not a billing system — it's a liability. Targets must be proportional to tenure. Jaicind's billing export to vendors depends on this being accurate.
- **Learning:** Any constant that is "per entity for a time period" is not actually a constant — it is a function. Encoding it as a literal is an architectural category error. Reject all hardcoded financial figures; derive them from first principles (working days × rate).

## [Entry 008] | Date: 2026-05-04
- **Pitfall:** **Anten/SipraHub LOP Logic Built Without Stakeholder Sign-Off.** The dashboard branch staged Migration 018 (`payroll_type` column) and a `test_payroll_segregation.py` suite encoding "1 holiday/month for Anten, 10+1 for SipraHub" LOP rules. On the May 4 call, Jaicind (verbatim): *"Don't go into that LOP and things... Let's not even look at it now. We only need to worry about what we are going to build our customer."* The customer-billing path was already hours-only, so the LOP work was both out-of-scope and risked being deployed accidentally.
- **Root Cause:** Verbal requirements from Senthil/Jaicind were converted into code before being mapped to a Pydantic schema or confirmed against the customer-billing contract. The work was speculative — solving for a payout/HR concern that the stakeholder explicitly does not want RMS to own.
- **Refactor:** (1) Migration 018 marked **STAGED — DO NOT APPLY** in its header and gated behind `SRE_CHECKLIST.md::G0.4`. (2) Billing engine path verified via `grep -rn "payroll_type|is_lop|LOP" backend/app/` returning zero hits in production code. (3) Replaced LOP responsibility with a thin Finance hand-off: `GET /api/exports/leaves-taken?month=YYYY-MM` returns raw `(employee_id, name, month, total_leaves_taken)` — Finance computes payout themselves. (4) Added `TestHolidayWorkedIsBillable` regression suite to prevent any future LOP/holiday branch from silently zeroing logged hours.
- **The "Why":** The May 4 transcript is the latest stakeholder directive and overrides any earlier sprint planning. Shipping LOP logic now would (a) violate a direct order, (b) entangle RMS in payout/HR scope it does not own, and (c) create a maintenance liability around Anten vs SipraHub policy rules that change quarterly.
- **Learning:** Verbal requirements must be matched to an explicit stakeholder-confirmed schema before any DB column is added. When in doubt, route the responsibility to the team that owns the policy (Finance) via a raw-data export rather than absorbing the rules into RMS.
